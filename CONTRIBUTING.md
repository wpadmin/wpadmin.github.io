# Принципы разработки

## Главная философия
**Простота и надежность превыше всего.** Код должен быть максимально простым, понятным и надежным. Никакого overengineering. Если джуниор не понимает код за 30 секунд - код плохой. Не усложняй без крайней необходимости.

---

## Фундаментальные принципы

### 1. KISS - Keep It Simple, Stupid
Всегда выбирай самое простое решение из возможных. Сложность - враг надежности. Простой код легче понять, протестировать, поддерживать и отлаживать. Если можешь решить задачу в 10 строк - не пиши 100.

### 2. YAGNI - You Aren't Gonna Need It
Не пиши код "на будущее". Добавляй функциональность только когда она реально нужна прямо сейчас. Не создавай абстракции для одного случая использования. Не делай систему расширяемой, пока не появилось хотя бы 3 реальных кейса.

### 3. DRY - Don't Repeat Yourself
Каждая часть бизнес-логики должна иметь единственное представление в системе. Но не путай дублирование кода с дублированием знания - иногда похожий код делает принципиально разные вещи и его не стоит объединять.

### 4. Единая ответственность (Single Responsibility)
Каждый модуль, класс, функция должны отвечать только за одну вещь и иметь только одну причину для изменения. Если в описании функции есть союз "и" - скорее всего, её нужно разбить на две.

### 5. Ранний возврат (Early Return)
Проверяй граничные условия в начале функции и выходи сразу. Это избавляет от лестницы вложенных if и делает основную логику плоской и читаемой. Guard clauses - твои друзья.

### 6. Явное лучше неявного
Код должен быть очевиден. Никаких магических чисел, неявных преобразований, скрытых побочных эффектов. Лучше быть многословным, но понятным, чем кратким, но загадочным.

### 7. Композиция лучше наследования
Предпочитай композицию глубоким иерархиям наследования. Это дает большую гибкость и меньшую связанность. Наследование создает жесткие зависимости, композиция - слабые.

### 8. Fail Fast - падай быстро
Если что-то пошло не так - узнай об этом немедленно. Не маскируй ошибки, не возвращай null или пустые значения. Лучше упасть сразу с понятной ошибкой, чем молча продолжить с некорректными данными.

### 9. Правило бойскаута
Оставляй код чище, чем он был до тебя. Увидел опечатку - исправь. Непонятное имя - переименуй. Неиспользуемый импорт - удали. Но не рефактори весь проект, когда делаешь маленький багфикс.

### 10. Принцип наименьшего удивления
Код должен работать так, как ожидает разумный разработчик. Функция get не должна ничего менять, функция calculate не должна сохранять в базу, метод toString не должен делать HTTP запросы.

## Правила написания кода

### Именование
- **Функции** - всегда глаголы: `calculateTotal()`, `getUserById()`, `validate()`
- **Переменные** - существительные: `userCount`, `totalPrice`, `connection`
- **Булевы** - вопросы: `isActive`, `hasPermission`, `canEdit`, `shouldUpdate`
- **Константы** - верхний регистр: `MAX_CONNECTIONS`, `DEFAULT_TIMEOUT`, `API_URL`

### Функции
- Максимум 20-30 строк, идеально 5-10
- Максимум 3 параметра, если больше - используй объект
- Делает одну вещь и делает её хорошо
- Не имеет побочных эффектов, если это не её прямое назначение

### Комментарии
- Объясняй ПОЧЕМУ, а не ЧТО делает код
- Документируй неочевидные решения и хаки
- Предупреждай о важных особенностях
- Удаляй устаревшие и неактуальные комментарии
- Код должен быть самодокументируемым

## Git и контроль версий

### Conventional Commits - СТРОГО
Все коммиты **только строчными буквами**, без восклицательных знаков, без тела сообщения:

```
feat: add payment integration
fix: correct price calculation
refactor: simplify user validation
docs: update readme
style: format code
test: add user tests
chore: update dependencies
```

### Правила коммитов
- **Атомарность** - один коммит = одно логическое изменение
- **Никаких подписей** - не пиши "by AI", "generated", имена разработчиков
- **Простота** - максимум 50 символов в заголовке
- **Только английский** - даже если проект русский
- **Настоящее время** - "add feature", не "added feature"

## Принципы надежности

### Валидация
- **Всегда валидируй** пользовательский ввод
- **Валидация на сервере** - клиентская только для UX
- **Fail Fast** - отклоняй некорректные данные сразу
- **Используй схемы** - Zod, Joi, JSON Schema

### Обработка ошибок
- Лови ошибки на правильном уровне абстракции
- Логируй достаточно информации для отладки
- Не глуши исключения без обработки
- Предоставляй понятные сообщения об ошибках

### Безопасность
- Никогда не доверяй внешним данным
- Используй параметризованные запросы для БД
- Не изобретай криптографию - используй проверенные библиотеки
- Принцип минимальных привилегий везде

## Антипаттерны - чего избегать

### God Object
Один класс/модуль, который делает всё. Разбивай на отдельные компоненты.

### Premature Optimization
Оптимизация до того, как это стало реальной проблемой. Сначала работающий код, потом быстрый.

### Magic Numbers
Числа в коде без объяснения. Всегда используй именованные константы.

### Over-engineering
Создание сложных абстракций для простых задач. KISS и YAGNI - твои мантры.

### Not Invented Here
Написание всего с нуля вместо использования проверенных решений.

## Правила производительности

1. **Не оптимизируй заранее** - профилируй, находи узкие места, потом оптимизируй
2. **Измеряй, не гадай** - используй реальные метрики, не предположения
3. **Правильные структуры данных** - часто это решает 90% проблем
4. **Кэшируй разумно** - но помни про инвалидацию
5. **Асинхронность где уместно** - но не усложняй без необходимости

---

## Рефакторинг

1. **Маленькие шаги** - один рефакторинг за раз
2. **Тесты первыми** - сначала покрой тестами, потом меняй
3. **Не меняй поведение** - рефакторинг не добавляет функциональность
4. **Частые коммиты** - коммить после каждого успешного изменения
5. **Откат, не починка** - если сломал, откатись к рабочей версии

---

## Итоговые правила

### Приоритеты (в порядке важности)
1. **Работает правильно** - код делает то, что должен
2. **Простой и понятный** - легко читать и модифицировать
3. **Надежный** - обрабатывает ошибки, валидирует данные
4. **Тестируемый** - можно покрыть автотестами
5. **Производительный** - оптимизирован где нужно

### Золотое правило
> "Сначала заставь работать, потом сделай правильно, потом сделай быстро" - Kent Beck

Именно в таком порядке. Не наоборот.
